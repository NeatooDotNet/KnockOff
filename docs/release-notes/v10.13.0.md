# v10.13.0 - Property Value Pattern Prioritization

**Release Date:** 2026-01-10

## Summary

This release improves documentation and skill guidance to prioritize the `Value` pattern over `OnGet` callbacks for configuring property stubs with static test data.

## Documentation Improvements

### Property Configuration Guidance

KnockOff properties support two configuration patterns:

| Pattern | Use Case |
|---------|----------|
| `stub.Property.Value = x` | Static values (recommended) |
| `stub.Property.OnGet = (ko) => x` | Dynamic/computed values |

**Before:** Documentation led with `OnGet` callbacks for all property configuration.

**After:** Documentation now recommends `Value` for static test data, reserving `OnGet` for dynamic scenarios.

## Examples

**Use `Value` for static test data (recommended):**
```csharp
knockOff.Name.Value = "John Doe";
knockOff.IsEnabled.Value = true;
knockOff.RetryCount.Value = 3;
```

**Use `OnGet` for dynamic behavior:**
```csharp
// Different value each call
var counter = 0;
knockOff.NextId.OnGet = (ko) => ++counter;

// Depends on stub state
knockOff.IsConnected.OnGet = (ko) => ko.Connect.WasCalled;
```

## Decision Guide

| Scenario | Use |
|----------|-----|
| Static test data | `Value` |
| Pre-populate before test | `Value` |
| Different value each call | `OnGet` |
| Depends on other stub state | `OnGet` |
| Computed from test context | `OnGet` |
| Throw on access | `OnGet` |

**Rule of thumb:** Start with `Value`. Only use `OnGet` when you need dynamic behavior.
